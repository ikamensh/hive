# Hive testing battle-test notes: 2026-06-17

Goal: run current Hive's testing workstream against Hive itself, improve the
process as needed, and keep going until Hive autonomously catches five real
issues.

## Running Observations

| Status | Observation | Why it matters | Suggested fix |
| --- | --- | --- | --- |
| Fixed in this branch | The runner advertised no `browser` capability even though Node Playwright was available through `npx --no-install playwright --version`; it only checked Python Playwright. | UI stories would stay `blocked_resources` on a repo whose frontend can already be driven by the installed Node tooling. | Detect Node Playwright CLI as a browser driver before capability probing. Added runner tests for this. |
| Candidate fix | Docker was installed but the daemon was not running, so the runner could not advertise `docker`. | The spec wants highest feasible fidelity, but local self-testing silently drops to browser/local-only capability unless the operator notices Docker/OrbStack is down. | Improve the capability todo text to name the exact failing Docker command/output and suggest starting Docker/OrbStack. |
| Candidate fix | `production_app` refuses local FileStore runtime, so an isolated local self-test requires managed Firestore/GCS plus a throwaway workspace id. | This is good for production safety but awkward for quick battle tests; the first attempted isolated `/tmp` chief failed before startup. | Add an explicit documented self-test recipe, or a clearly named test-only app factory/CLI mode that uses FileStore and cannot be mistaken for production. |
| Fixed in this branch | A runner auto-probes every detected backend. For a focused Codex run, I had to narrow `PATH` manually so Claude/Cursor/Gemini did not create extra probe traffic. | Multi-backend machines are normal, but a single-campaign test often wants one backend to control cost and noise. | Added `HIVE_RUNNER_BACKENDS=codex,cursor,...` so registration/probing can be limited without PATH surgery. |
| Candidate fix | There is no obvious CLI command for creating/testing episodes, so I used raw API calls with `curl`. | The testing workstream has UI/API routes, but a repeatable operator battle-test should be scriptable without memorizing endpoints. | Add `hive test-episode` / `hive test-refresh` CLI parity commands, mirroring issue-run commands. |
| Watch | Live self-testing needs a dedicated workspace id to avoid colliding with the real Hive chief's leader lease and state. | This is easy to get right once remembered, but easy to forget under pressure. | Include workspace id guidance in the self-test recipe and have startup print it prominently. |
| Candidate fix | The Hive repo has no `wiki/infrastructure.md`, but the testing prompt tells sweep agents to use the project's run instructions. | A self-test agent can still infer `uv run ...` and `npm run dev`, but missing official run instructions makes sweep quality more variable and encourages source spelunking. | Add a concise `wiki/infrastructure.md` with chief, runner, web, and testing-safe local run recipes. |
| Fixed in this branch | After creating the testing episode, the refresh task sat pending for a noticeable delay before dispatch. | Episode creation should feel alive; waiting for a periodic supervisor tick makes the UI/API look stuck even when capacity is ready. | Wake the supervisor from `create_test_episode` / `test-refresh` after queueing the refresh task. |
| Candidate fix | The refresh task reported 358k input tokens to conclude no backlog changes were needed, while `cost_usd` remained `0.0`. | A no-op refresh should be cheaper, and zero-dollar runner accounting makes the project budget less trustworthy for subscription/CLI backends. | Add cheaper refresh prechecks/digests for unchanged specs, and improve runner cost accounting or at least surface "unpriced tokens" separately. |
| Candidate fix | Parallel sweeps independently started local services; Vite happened to auto-increment to ports `5173`, `5174`, and `5175`, while chief attempts used `8080`. | The episode relies on tool defaults to avoid port collisions. That is fragile and can produce false environment failures. | Give each test task an explicit `HIVE_TEST_PORT_BASE` / allocated port bundle in its instructions and environment. |
| Fixed in this branch | A sweep can return `SWEEP: BLOCKED` / structured `outcome: blocked`; Hive marked the story blocked but did not file a human todo or question with the reason. | Blocked testing is exactly when the operator needs actionable setup instructions; silently blocked stories reduce the value of the episode. | Escalate blocked sweep results with story, task, summary, output, and artifacts. |
| Candidate fix | My first self-test runners did not include `HIVE_GCP_PROJECT` / `HIVE_GCS_BUCKET`, so a sweep trying `hive run` blocked on missing managed-state env. | Self-testing Hive needs the runner task environment to be able to boot Hive, not only the outer chief. | Include managed-state env in the documented self-test runner command, or pass project/test env bundles explicitly to test tasks. |
| Candidate fix | A sweep exercised the frontend with `VITE_MOCK=1` and reported `fidelity: local`. | Mock UI can find real UI defects, but it is weaker than local app plus real backend. The fidelity field currently cannot distinguish those. | Extend fidelity or result metadata with `mock`, `local_backend`, `managed_backend`, and `docker` so issue filing can label evidence strength honestly. |
| Candidate fix | The artifact uploader recursively uploaded a temporary Playwright `node_modules` tree under `.hive/artifacts/browser-runner`. | Evidence upload became noisy and slow, and issue bodies can end up with irrelevant dependency files. | Exclude common dependency/build/cache directories (`node_modules`, `.venv`, `dist`, caches) under `.hive/artifacts`, or enforce an allow-list of evidence file extensions. |
| Candidate fix | After findings were suspected, newly added runner capacity picked up remaining sweep tasks before all reproduction tasks. | Confirmation is the trust gate; delaying it means the episode can spend a lot on broad sweeps before learning whether earlier findings are real. | Prioritize `test_reproduce` / `test_judge` ahead of additional `test_sweep` tasks once findings exist, or reserve a confirmation lane. |
| Fixed in this branch | Managed-env runners failed checkout with GitHub `Duplicate header: "Authorization"` during `git fetch origin`; the second clean run proved filtering inherited `GIT_CONFIG_*` env was insufficient because global GitHub CLI config can also provide an extraheader. | A runner with both GitHub credential helper auth and an injected extraheader can fail before any testing happens. | Reset GitHub's multi-valued `http.https://github.com/.extraheader` list before adding Hive's token header, and keep inherited non-GitHub Git config entries. |
| Fixed in this branch | Issue `#10` was filed with a huge evidence section because Hive merged every uploaded task artifact into the finding, even though the finding named only two evidence files. | Issue bodies become unreadable and can expose irrelevant generated files. | Use explicit per-finding `evidence_blobs` as the source of truth; fall back to all task artifacts only when the finding names none. |
| Fixed in this branch | Reproduction tasks uploaded artifacts from earlier tasks in the same reused checkout. `git clean -fd` preserved ignored `.hive/artifacts`, so evidence bled across task boundaries. | Confirmations should be independent. Stale screenshots/logs can pollute issue evidence and make "reproduced from scratch" less trustworthy. | Clear runner-managed `.hive/artifacts`, `.hive/result.json`, and `.hive/issue-*` scratch state before every task.
