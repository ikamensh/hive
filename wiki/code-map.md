# Code map (per-file detail)

The folder-level map and conventions live in `AGENTS.md` (auto-loaded). This file is the per-file detail ŌĆö read it when navigating into a subpackage. Architecture rationale (the "why") is in `architecture.md`.

- `hive/models.py` ŌĆö pydantic domain models; persisted as dicts via the store. `Plan`/`PlanItem` are the iteration plan (wiki/iteration-plan.md): the item is the durable work unit itself ŌĆö review states (`proposed/approved`) then the pipeline's execution states on one record, `PLAN_ITEM_PARKED/IN_FLIGHT/TERMINAL` the shared groupings. `Subscription` (AI plans the user owns) and `HumanTask` (operator todos with markdown instructions, e.g. CLI logins on remote runners) are org-level, not project-scoped. Multi-user: members are `admin` (default) or `resource_provider` (`WorkspaceMembership.role` ŌĆö lends capacity, no edit rights, manages only what they own); `Machine.owner_user_id`/`Subscription.owner_user_id` say whose hands/license it is, and `HumanTask.assignee_user_id` routes a todo to the one user who can act (login and dark-machine escalations set it from the machine owner). `OrchestratorRun` records one planner invocation's token usage + cost (the chief side of the bill; runner spend lives on `Task`). `IssueItem` (collection `workstreams`) is one GitHub issue mirrored as a pipeline work item; `Directive` is a human-authored ask routed by Hive (filed as a GitHub issue, tracked to done by the issue pipeline) and `Checkout` records a repo's working copy on one machine with git-drift facts (`drift()` = ahead or dirty) ŌĆö see `wiki/project-launchpad.md`.
- `hive/agents/usage.py` ŌĆö license-usage introspection: `parse_reset_hint` pulls an explicit reset time out of rate-limit error text (epoch-after-pipe, ISO-near-anchor, durations, local clock times, weekdays, gemini `retryDelay`) in the machine's own timezone; `collect_usage(backend)` reads the provider's own gauge without spending quota (claude: Claude Code OAuth token ŌĆö keychain on macOS, `~/.claude/.credentials.json` elsewhere ŌĆö authorizes `GET api.anthropic.com/api/oauth/usage` for exact per-window used-percent + reset; codex: the newest `rate_limits` snapshot in `~/.codex/sessions/**/rollout-*.jsonl`; cursor/gemini-cli: nothing native). Both gauges are account-wide, so the same login on two machines converges without cross-resource linking. Snapshots travel as plain dicts: the runner's `execute()` attaches one per task result plus a `reset_at_hint` when exhausted, and the register heartbeat carries all backends' snapshots refreshed every `USAGE_REFRESH_S`. `python -m hive.agents` prints what a machine can collect.
- `hive/_control/allowances.py` ŌĆö per-project agent allowances (`Project.agent_grants`, design `wiki/agent-allowances.md`): pure functions assigning today's dispatched sessions to grants (unlimited first, then most headroom) and answering `admits`/`remaining`/`resolve_agent`. Count-based because subscription CLIs report ~zero `cost_usd`, so the money budget can't meter them. Enforced hard in `Supervisor.dispatch` (per task; exhausted pairs roll the project to `blocked_budget`), advisorily in the planner snapshot (`AGENT ALLOWANCE` line), and by remapping in the deterministic pipelines (`issues._make_issue_task`, `testing.queue_refresh_task`/`start_episode`, `intake.trusted_capacity`). Probe/preflight are exempt. Surfaces: `PATCH /api/projects/{id} agent_grants` (validated against `BACKEND_NAMES`), `hive set --grant 'claude:5/day'` (repeatable), `allowance` in the project payload.
- `hive/_control/limits.py` ŌĆö chief-side consumer: `apply_snapshot` writes the freshest snapshot onto `Resource.usage_*` and syncs `cooldown_until` with it (a window Ōēź98% cools the resource until that window's own reset; a healthy snapshot clears only snapshot-made cooldowns, prefix `usage-window:`); `cooldown_after_exhaustion` picks the retry moment for a rate-limited task (message hint > spent window > soonest rollover > flat hour); `record_snapshot`/`record_exhaustion` append `LimitEvent` history (snapshots only on material movement); `estimate_window_budgets` learns tokens-per-window from percent deltas between snapshots (median across pairs; lower bound because non-Hive usage moves the gauge for free); `limits_view` powers `hive show limits` / the `limits` section of `GET /api/show`.
- `hive/agents/` ŌĆö **standalone package** (see `wiki/subsystems.md`): everything about the coding-agent CLIs on a machine. `backends.py` is the single source of truth: the `REGISTRY` of `Backend(name, make_session)` records, derived `BACKEND_NAMES` (the orchestrator's allow-list, the runner's capability filter), the `make_session(backend, model)` dispatcher, discovery, failure classification (`classify_failure`), and the usability-probe prompt (`PROBE_MARKER`, `probe_instructions`) ŌĆö add a backend = one entry here; session factories import `kodo` lazily so the chief imports the registry without kodo installed. `results.py` is the generic structured-result machinery (`ResultSpec` around any pydantic model, `call_agent` with the warm-session repair loop, `AgentCallResult`); hive's per-TaskKind contracts stay in `hive/runner/_agent_results.py`. `run.py` is `run_agent()` ŌĆö the one-call session+loop+result lifecycle both the runner daemon and scripts use (`on_session` hook = cancel channel, session handle on the result). `probe.py` builds the throwaway local probe repo and validates marker+clean-tree (`probe_backend` is the standalone composition).
- `hive/_workstreams/issues.py` ŌĆö internal implementation of GitHub issue solving, a deterministic per-issue pipeline inside normal projects with **no orchestrator/LLM in the path** (full design in `wiki/issue-solving.md`, product model in `wiki/unified-project-work.md`). GitHub I/O (`fetch_open_issues_full` = issues+comments+embedded-image-URLs, `merge_branch` via the merges API, `resolve_issue_on_github`, authed like `github_repos.py`) is kept separate from the pure store ops (`reconcile`, `advance_issues`, `create_review_task`) so the latter test without network. Flow: a human-triggered sync/run calls `reconcile` (open issue ŌåÆ issue work item status `queued`, full doc in `description`, image URLs in `issue_attachments`; cancel externally-closed; re-gate any still-open non-`done` issue with no live task ŌĆö `blocked_clarity`/`rejected`/reopened/errored-mid-flight ŌĆö back to `queued`), then `advance_issues` starts **strictly one issue at a time per GitHub issues workstream**: if none is in flight (`resolving`/`reviewing`) it promotes the lowest-`order` selected `queued` issue to `resolving` and queues one resolve task on branch `hive/issue-<n>`. `advance_issues` is called again after every issue-task landing, so the next issue branches from a default branch that already has the prior fixes. The resolve agent (`prompts/resolve.md`) clarifies then fixes in the same warm session, ending `OUTCOME: FIXED|BLOCKED`. `hive/runner/_task_results.py` parses it (`parse_resolve`): BLOCKEDŌåÆ`blocked_clarity`, FIXEDŌåÆ`reviewing` + queue a review task (`prompts/review.md`, fresh agent, fix-on-spot, ends `REVIEW: ACCEPT|REJECT`). `parse_review`: REJECTŌåÆ`rejected` (agent commented), ACCEPTŌåÆ`TaskResultProcessor._land_review` merges the branch into the default branch (merges API, no PR) and closes the issue ŌåÆ `done` (merge/close failure escalates a `HumanTask`). The runner materializes `.hive/issue-<n>/` (ISSUE.md + downloaded attachments, git-excluded) before the agent runs (`runner.prepare_issue_workspace`). The planner sees issue work as read-only project context; deterministic issue tasks advance from scan/run and task-result handling.
- `hive/_workstreams/plans.py` ŌĆö iteration plans, the doc-fed resolveŌåÆreviewŌåÆmerge pipeline (design `wiki/iteration-plan.md`). A `Plan` is the ordered `PlanItem` list for one iteration; the item **is** the work item the pipeline executes (one record: review states `proposed/approved` owned by the plan UI, then the pipeline's execution states). Store ops: `create_draft` (replaces an unapproved draft, never an approved plan), `approve_item`/`approve_all` (the flip; on a live plan an amendment flip queues directly), `activate` (all-items-approved gate ŌåÆ queue everything in store, commit the plan doc to the spec home best-effort with an escalate on failure, start the first item), `advance_plan` (strict order, one in flight, **parked items stall the queue** ŌĆö a blocked/rejected item is the human's decision point), `retry_item`/`cancel_item`/`abandon_plan`, `refresh_plan` (all-terminal ŌåÆ complete). Execution tasks are `TaskKind.resolve`/`review` with the item document embedded in the instructions (`build_work_doc`; prompts `plan_resolve`/`plan_review` ŌĆö report replaces GitHub comments, BLOCKED/REJECT reasons land in `parked_reason` + Needs-you); `work_item_id` pointing at a `PlanItem` is how `TaskResultProcessor` routes results (`_handle_plan_task_result`: FIXEDŌåÆreview, ACCEPTŌåÆ`merge_branch` to the default branch + delete branch ŌåÆ done ŌåÆ advance; landing failure escalates with the `plan_item_done` predicate; plan completion wakes the planner to propose the next iteration). No GitHub issues/PRs in the loop ŌĆö GitHub is a source of work in, never hive's internal ledger.
- `hive/_workstreams/testing.py` ŌĆö internal testing-workstream implementation (full design in `wiki/testing.md`): mirror `acceptance/*.md` into `Story` rows (`reconcile_story_backlog`), snapshot a `TestEpisode` and queue refresh/sweep/confirm tasks, denoise sweep findings, and file confirmed ones as GitHub issues (`file_or_update_finding_issue`/`close_story_issue`, kept apart from the store ops like `issues.py`). `story_health` is the deterministic backlog verdict shared by UI/CLI/API (`testing_health` in the project payload): a missing backlog or weak stories (`story_quality_problem` ŌĆö no intent / no concrete examples) yields Hive's standing offer to draft or rewrite them autonomously (the refresh task); untested/failing points at running an episode. `auto_testing_action` turns that verdict into the autonomous tick's decision (see the supervisor's `testing_check` poller): only for `testing_auto` projects with a positive daily budget, nothing in flight, and a per-kind daily cooldown from store facts. CLI: `hive test-refresh`, `test-run`, `stories`, `test-cancel`; `hive set --testing-auto`.
- `hive/_workstreams/testability.py` ŌĆö the per-project testability contract (design `wiki/testability-contract.md`): mirror the spec home's `testability.md` into a `TestabilityContract` record (`reconcile_contract`; probe evidence survives only while the content digest matches), turn the draft agent's structured `decisions` into deduped `Question`s (`dedup_key = testability:<ws>:<key>`; asked-once semantics, options + recommendation in the text), queue `testability_draft`/`testability_probe` tasks, and compute `testability_health` (missing ŌåÆ drafting ŌåÆ decisions ŌåÆ draft ŌåÆ probing ŌåÆ broken ŌåÆ verified) with Hive's standing offer. Draft completion auto-chains the probe (`_task_results._handle_testability_draft_result`); an answered decision auto-queues a re-draft (`clarifications._fold_testability_answer`); sweeps/confirms embed the contract via `contract_context`. Auto-episodes gate on a verified contract (`testing.auto_testing_decision`).
- `hive/_workstreams/ci.py` ŌĆö internal CI auto-fix implementation (full design in `wiki/ci-autofix.md`). `fetch_ci_status` reads a repo's default-branch CI (check-runs + legacy commit-status APIs) into one verdict (`passing|failing|pending|none`); `check_and_autofix` files a deduped `hive-ci` GitHub issue when red (marker `<!-- hive-ci sha=ŌĆ” -->` in the body, so the same red commit is never refiled) and reuses `issues.reconcile`/`advance_issues` to queue the fix ŌĆö no separate fixer. Gated by `Project.ci_autofix`, three triggers run the same path (all idempotent via per-sha dedup): the **webhook** `POST /api/ci/webhook` (bearer-authed by `HIVE_GITHUB_WEBHOOK_SECRET`, fed by `deploy/ci-autofix.github-workflow.yml` ŌĆö a GitHub Actions forwarder modelled on the `ci-assistant` VPS service; forwards the failing-run logs as `details`); the **supervisor poll** every `CI_CHECK_INTERVAL_S` via an injected `ci_check` callback (wired in `production_app`); and the **manual** `POST .../workstreams/{id}/check-ci` (CLI `hive check-ci`, web "check CI" button).
- `hive/_workstreams/preflight.py` ŌĆö internal issue-solving preconditions, so a misconfigured run fails up front instead of mid-pipeline. `preflight_checks` (chief): `repo_set`, `gh_token_present`, `repo_write_access` (the token's `permissions.push`, needed for merge+close), soft `issues_enabled`/`codex_runner_usable`. `create_preflight_task` queues a `TaskKind.preflight` runner self-check that `runner.run_preflight` executes against the real repo (push a throwaway branch + delete it, `gh auth status`) ŌĆö the agent-facing git/gh auth the chief can't see. Workstream-aware `/workstreams/{id}/preflight` is the primary route; `POST /api/projects/{id}/issues-preflight` (CLI `hive preflight`) remains a compatibility wrapper for the spec repo.
- `hive/_control/show.py` ŌĆö subsystem introspection behind `hive show` / `GET /api/show`, so each subsystem is testable in isolation instead of only through the full autonomous loop: `machines` (durable machines, where they live, online/dark, which hosts the chief), `agents` (every launchable (backend, machine) pair with actionable notes), `subscriptions` (expectation vs reality: where each license serves, `login_needed` gaps, `unregistered` proven-but-unrecorded capacity via `subscription_candidates`, `unowned` backends), and `autonomy` (the supervisor's recurring jobs ŌĆö ci_check, testing_check per repo, dark-machine watch ŌĆö with period, gate reasons, and what each would do on which machines if it resolved now, via `testing.auto_testing_decision`). Read-only over the store; `spend_today` injected like `overview.py`.
- `hive/_control/escalation.py` ŌĆö the "ask a human for help" primitive and its self-closing lifecycle (G23ŌĆōG26). `escalate(...)` files a typed `HumanTask` (kind access/infra/repair/env/external), idempotent by `dedup_key` (a stable condition identity like `access:codex:hive-vm` shared by every producer) with (title, scope) as the keyless fallback. Most todos carry a `resolution` predicate ŌĆö a store fact that proves the condition is gone (`resource_usable`, `machine_online`, `task_inactive`, `story_verdict`, `finding_decided`, `workstream_done`, `conversation_done`, `orchestrator_ran`, `project_state_not`); `resolve_open_todos` sweeps them on every supervisor step *and* after each task result, recording the evidence in `resolved_reason`. `resolve_todo(dedup_key)` is the event-driven close for conditions without a plain store fact (intake retry success, test-refresh finalization). Only predicate-less todos (external kind) need a manual done.
- `hive/_control/todo_triage.py` ŌĆö the AI second opinion on the open todo board, the backstop for what escalation's deterministic layer cannot see: differently-worded duplicates for one root cause and predicate-less todos whose condition the store facts show is gone. One cheap LLM call over (open todos + fleet facts) proposes `keep|duplicate|stale`; guards are code-enforced (close-only, predicated todos untouchable, never trade a self-closing todo for a manual one). Transport is an injectable `prompt -> text` callable (`api.make_todo_triage` wires the configured adapter); the supervisor runs it only when the set of open todos changed, at most once per `TODO_TRIAGE_INTERVAL_S`.
- `hive/_control/clarifications.py` ŌĆö internal answer/dismiss handling for parked clarification questions, behind `POST /api/questions/{id}/answer|dismiss`. `apply_answer` persists the answer, best-effort appends the raw text to the spec repo's `input-log/` (escalating a `HumanTask` on write failure via `can_write_spec_repo` gating), and wakes the planner to distill it; `dismiss` records the dismissal and nudges any parked workstream. Intake's sibling (both write durable spec material); free functions over store/config/supervisor, no FastAPI (the 404 guard stays in the route).
- `hive/_control/intake.py` ŌĆö internal project intake: the trusted-scout conversation that aligns a project with the user and pushes durable spec files before planning. Free functions over `store`/`config`/`supervisor` (`create_conversation`, `writable_conversation`, `queue_turn`, `accept`, `is_done`, plus `trusted_capacity`/`spec_status`/`prompt`), called by the `/intake/*` routes; raises `HTTPException` directly (like `_integrations/auth.py`) so the routes stay thin. `TRUSTED_SCOUTS` is the preference-ordered allow-list (codex gpt-5.5, claude opus). Design in `wiki/project-intake.md`.
- `hive/persistence/store.py` ŌĆö `StoreBase` (ABC) is the contract; `MemoryStore` (tests), `FileStore` (local JSON under `HIVE_DATA_DIR/store`), and `FirestoreStore` (prod) are independent implementations. `update(model, id, mutate)` is the atomic read-modify-write ŌĆö the way to mutate a contended doc (dispatch claim, result recording, resource counters, reboot requeue) without losing a concurrent writer: MemoryStore/FileStore guard every op with an RLock, FirestoreStore runs `mutate` in a transaction. `list(..., limit=N)` returns the most recent N (oldestŌåÆnewest) so snapshot/`get_project` reads stay bounded (Firestore still sorts/slices in Python; pushing the slice down would need composite indexes). Plain `put` stays last-write-wins for fresh/uncontended writes. `CachedStore` is the chief's write-through cache over the backing store: the leader lease makes the chief the single writer, so all reads come from process memory and only writes reach Firestore (without it, the supervisor tick + runner poll/heartbeat loops cost ~1.5M idle Firestore reads/day); leases and org context pass through uncached so fencing stays authoritative in the backend. Standalone package: documents are *any* pydantic models with an `id` ŌĆö collection names derive from class names (`collection_name`, `__collection__` override), a missing key matches the model's declared static default (the migration shim, generalized), and `CachedStore` hydrates lazily per collection so no model registry exists. `hive/config/storage.py` is the hive glue: selects the backend from `Config` (wrapping Firestore in `CachedStore` for the chief), exposes storage introspection, and copies a local store + blobs (`hive.models.ALL_MODELS`) to GCP via `POST /api/storage/export`.
- `hive/_control/supervisor.py` ŌĆö internal deterministic layer: `compute_state` (pure, backend-aware ŌĆö pending work whose backend no online runner offers is `blocked_resources`, not fake `working`) and `state_reason` (the one human sentence every surface pairs with the badge: cause + fix), dispatch (serialized per repo; skipped when over the project's daily budget ŌåÆ `blocked_budget`), orphan failing, asyncio loop waking the orchestrator on events (incl. a replan nudge when work is stuck on an unavailable backend). `spend_today` sums both finished `Task.cost_usd` and today's `OrchestratorRun.cost_usd`, so a budget caps the planner's own LLM spend too ŌĆö and the loop skips invoking the orchestrator entirely while over budget (it costs money), going quiet until the UTC-midnight rollover. Single-instance guard: a leader lease in the store (`claim_leader`, Firestore doc `settings/leader_lease`) is acquired at startup and renewed every tick, so a second chief on the same Firestore refuses to boot (and a fenced-out one exits). Two injected per-project pollers ride the loop (wired in `production_app`, `None` in tests): `ci_check` (CI auto-fix, `CI_CHECK_INTERVAL_S`) and `testing_check` (autonomous testing for `testing_auto` + budgeted projects, `TESTING_CHECK_INTERVAL_S` ŌĆö acts via `testing.auto_testing_action`). `check_dark_machines` rides every step too: a machine that heartbeated recently but has been silent past its availability-class threshold (`MACHINE_DARK_AFTER_S`: laptops 24h, servers 4h) gets an org-wide `HumanTask` with restart instructions, auto-closed on reconnect; machines silent past `MACHINE_RETIRED_AFTER_S` (7d) are treated as retired and stay quiet.
- `hive/runner/_task_results.py` ŌĆö private runner-result workflow boundary. `TaskResult` is the HTTP/result contract; `TaskResultProcessor` records completion, parses structured/legacy outcomes, accounts resource spend/quota, updates probe/intake/issue/testing state machines, performs deterministic issue landing, and wakes the supervisor only for planner-owned build/verify tasks. `hive/api.py` delegates `POST /api/tasks/{id}/result` here instead of owning workflow transitions in the route layer.
- `hive/runner/registration.py` ŌĆö the runner-protocol *request side*: `register(store, body, workspace_id)` is the heartbeat brain behind `POST /api/runners/register` (machine+runner upsert, reboot requeue of dropped work via `_requeue_dropped_work`, per-backend `Resource` reconciliation from discoveries/capabilities, startup probes, checkout git-facts via `upsert_checkouts`). `RunnerRegister.owner_user_id` (from `HIVE_RUNNER_OWNER`, written by `hive enroll`) claims an unowned machine for that member on first register ŌĆö never steals an owned one. Owns the `RunnerRegister`/`BackendDiscoveryInput`/`CheckoutReport` schema and the `PROBE_REPO` sentinel. `queue_probe`/`should_auto_probe`/`active_probe_task` are the probe primitives ŌĆö `queue_probe` is shared with the web `POST /api/resources/{id}/probe` route. Raises `HTTPException` directly (like `_integrations/auth.py`) so the routes stay thin.
- `hive/llm/` ŌĆö public facade for unified LLM access shared by every model-using flow; implementation files are private (`_core.py`, `_openai.py`, `_gemini.py`, `_provider.py`, `_parsing.py`, `_pricing.py`, `_model_intel.py`). `_core.py`: `ToolSet` (tool schemas from a callable's signature + docstring, and call dispatch), the `LLMAdapter` protocol (stateful `start`/`step`/`add_tool_results` conversation), and `ToolLoop` (provider-agnostic driver returning a `LoopResult` with the final text + summed `Usage` token counts both adapters report). `_openai.py`/`_gemini.py`: thin per-provider adapters ŌĆö OpenAI is the manual chat-completions loop (+ model auto-select); Gemini drives `google-genai` function calling manually (auto-calling disabled) so the same loop owns the cycle. `_provider.py`: `build_adapter(config)` + `resolve_provider`. `_parsing.py`: `extract_json` (also used by critique). Adding a provider = one private adapter plus a facade export only if callers need it.
- `hive/_control/orchestrator.py` ŌĆö the **planner**: an LLM tool-loop that proposes and maintains the iteration plan but never executes (design `wiki/iteration-plan.md`). `_generate` builds the configured adapter (`HIVE_ORCH_PROVIDER=auto|openai|gemini`, `HIVE_ORCH_MODEL` optional) and runs `ToolLoop(MAX_REMOTE_CALLS)` over `ToolSet(Tools.functions())`; `_build_adapters` is the test-injection seam. Each `invoke` records an `OrchestratorRun` (`_record_cost`, priced via `hive.llm._pricing`) so planner spend is visible and budgeted. Tool surface: `propose_plan(goal, items_json)` / `amend_plan(items_json)` (drafts and amendment proposals ŌĆö nothing executes before the human's flip), `ask_user` (structured markdown enforced), `withdraw_question`, `commit_to_spec`, `create_human_task` (guarded kinds + system dedup keys), `mark_goal_complete` ŌĆö rejected unless the latest plan is `complete` (every item merged or human-cancelled), nothing pending/running, no open questions; the pipeline's fresh-agent review per item is the structural quality gate. The predictability invariant lives in this split: the planner has no tool that queues, cancels, or completes execution. No `from __future__ import annotations` in this file ŌĆö `ToolSet`/google-genai schema inference inspects the `Tools` methods' runtime annotations. Conversation history persists to the blob store; cold start is always safe because every invocation gets a full state snapshot (incl. the plan + items) + spec digest.
- `hive/llm/_pricing.py` ŌĆö token-price estimates (USD/Mtok, longest-prefix match) that turn `OrchestratorRun` token counts into dollars; unknown models cost 0 (tokens still recorded). Refresh occasionally.
- `hive/_integrations/specrepo.py` ŌĆö internal shallow clone of the project's spec-home repo; digest for context (`digest_dir` works on any local dir), small commits for wiki/input-log distillation.
- `hive/_workstreams/critique.py` ŌĆö internal spec critique implementation (wiki/spec-critique.md): parallel critics (every model x tester/builder/consistency lens) + adjudicator run by the smartest model per `hive.llm._model_intel` (Artificial Analysis index estimates). LLM transport is a `prompt -> text` callable (kept separate from the `hive/llm` adapters because `scripts/spec_critique.py` runs it via CLI agents, not API keys); JSON is parsed with `hive/llm`'s shared `extract_json`. Prompts: `hive/llm/prompts/critic.md`, `adjudicator.md` (the "intake" role).
- `hive/api.py` ŌĆö FastAPI: web API (unauthenticated; sits behind a tunnel/Tailscale) + runner protocol (shared token via `X-Hive-Token`). `create_app` takes the blob store so the runner can upload per-task JSONL traces (`POST/GET /api/tasks/{id}/trace`). `production_app()` wires env config; SPA fallback serves `web/dist`. Project create takes name + `spec_text`; `PATCH` configures repos/toggles; planning wakes from intake acceptance (no manual start step). Project payloads carry `state_reason` and `spend_today`.
- `hive/cli.py` ŌĆö `hive` console script: full web-API parity. Glance surfaces (`projects`, `project`, `inbox`, `show`) print readable summaries by default with `--json` for scripts; everything else stays JSON. `hive new` is the one creation path (create + repo + budget + intake in one command); `hive intake <project>` is the one intake verb (show/-m/--proceed/--approve); workstream-scoped commands resolve the workstream id themselves when the project has exactly one of the kind (`WS_KIND_BY_COMMAND`/`resolve_workstream`); `--grant 'backend[=model]:N/day'` is the allowance syntax (`parse_grants`). `run(args, client)` takes any httpx-compatible client, so tests drive it with `TestClient` (`tests/test_cli.py` replays the whole loop CLI-side). `hive run`/`hive config` are the exceptions ŌĆö they act locally instead of calling the API. The **client target** (which remote chief + how to auth) is `resolve_targets(env, stored)` over `CLIENT_KEYS` (`HIVE_URL`, basic-auth `HIVE_BASIC_AUTH`, bearer `HIVE_TOKEN`): an explicit URL (env > stored, the inverse of `prepare_run_env`'s server precedence) names one chief; with none configured the CLI discovers ŌĆö localhost first (`hive run` dev loop), then the chief this machine's installed runner reports to (`runner_env_target`, `~/.config/hive/runner.env` incl. its perimeter creds) ŌĆö `main` walks the candidates on connection errors only (an HTTP status means a chief was found). These keys are kept out of the server env on `hive run`. `hive whoami` resolves the target + hits `/api/auth/me` (the "check auth" step); `main` turns 401/403 into an actionable credentials hint instead of a traceback. Testing: `hive test-refresh` drafts/aligns stories, `hive test-run` starts an episode, `hive stories` is the coverage view (health + offer per workstream), `hive test-cancel` stops an episode (peers to `hive scan` for issues). `hive project <id>` is the project detail; `hive show [machines|agents|autonomy]` is the org-level subsystem introspection (`_control/show.py`). `hive enroll --url <chief> --token <t>` is member self-onboarding: any member (incl. resource providers) mints the one-hour token on the web machines page ("add a laptop"), and the command exchanges it (`POST /api/enroll`) for runner credentials ŌĆö no GCP access needed ŌĆö then installs the launchd service via `deploy/install_mac_runner.sh` (non-mac: writes `runner.env` and prints how to start `python -m hive.runner`); the enrolled machine registers with `HIVE_RUNNER_OWNER` and is claimed for the minting user, so its login todos route to them. `hive login <backend> --machine <name>` fixes a machine-bound agent login on a remote runner: `LOGIN_RECIPES` + `login_ssh_argv` open an SSH session as root on the VM (TTY, codex's OAuth callback port forwarded) so the browser part stays on the operator's machine, then it probes the resource to prove the login (auto-closing its 'Fix login' todo). `hive run` boots the chief (uvicorn factory); `prepare_run_env` resolves tokens with precedence stored-config > ambient-env > `gh auth token` autodetect and prints provenance. `hive config set/unset/show/import` manages hive's own token store (`~/.config/hive/config.env`, chmod 600, keys allow-listed in `CONFIG_KEYS`) so the operator can give hive separate keys (e.g. separate cost tracking); stored values override ambient env on `run`, and `import` seeds the store from autodetected tokens.
- `hive/runner/_daemon.py` ŌĆö private daemon implementation behind `python -m hive.runner`, now a composition over `hive.worker`: it supplies the register payload (backends/discoveries/capabilities/checkouts/usage snapshots) and `execute()` (checkout the task's repo/branch ŌåÆ `hive.agents.run_agent` ŌåÆ upload the kodo JSONL trace + artifacts), while `WorkerLoop` owns connect/heartbeat/poll/report. Chief discovery is `hive.worker`'s roster: `HIVE_URL` seeds candidates, every register response merges the chief's `HIVE_ADVERTISED_URLS`, and after 3 consecutive errors the loop re-resolves ŌĆö a relocated chief only has to advertise its new address. With `HIVE_RUNNER_SELF_UPDATE=1` (dedicated service clones only) the `between_tasks` hook exits when origin/main moves ahead; the service wrapper pulls and respawns. The `before_poll` hook drains on the operator pause flag (see `control.py`), and the daemon mirrors what it's doing into the local status file (idle/task/paused) for the menu bar app.
